     config.set('mode', mode);
   }

   export function getModels() {

     config.set(`models.${tier}`, modelId);

  <action>
Create `lib/profile.js` implementing shell profile detection, reading, writing, and idempotent marker-block injection per D-01, D-02, D-03, D-04.

The module must have these exact exports:

1. `getProfilePath()` -- Detect the correct shell profile path:
   - On `win32`: Run `powershell -NoProfile -Command "echo $PROFILE"` via `execSync` to get the live PowerShell profile path. If that fails, fallback to `path.join(os.homedir(), 'Documents', 'PowerShell', 'Microsoft.PowerShell_profile.ps1')`.
   - On macOS/Linux: Check `process.env.SHELL`. If it contains `zsh`, return `path.join(os.homedir(), '.zshrc')`. Otherwise return `path.join(os.homedir(), '.bashrc')`.

2. `readProfile(profilePath)` -- Read the profile file content as UTF-8 string. If the file does not exist, return an empty string (do NOT throw ENOENT).

3. `writeProfile(profilePath, content)` -- Write content to the profile file. Create parent directories with `{ recursive: true }` if they do not exist (handles the case where PowerShell profile directory doesn't exist yet per Pitfall 3).

4. `injectBlock(profileContent, block)` -- Idempotent marker-block injection:
   - Use markers: `# BEGIN claude-switcher` and `# END claude-switcher`
   - Regex: `new RegExp('# BEGIN claude-switcher[\\s\\S]*?# END claude-switcher\\n?', 'm')`
   - If the marker block already exists in `profileContent`, replace it with the new block.
   - If no marker block exists, append `\n# BEGIN claude-switcher\n{block}\n# END claude-switcher\n` to the end.
   - Return the new profile content string (does NOT write to disk -- caller does that).

5. `removeBlock(profileContent)` -- Remove the marker block entirely:
   - Regex: `new RegExp('\\n?# BEGIN claude-switcher[\\s\\S]*?# END claude-switcher\\n?', 'm')`
   - Return the profile content with the block removed.

6. `buildOpenRouterBlock(apiKey)` -- Generate the env var block for the detected platform:
   - On `win32` (PowerShell syntax):
     ```
     $env:ANTHROPIC_BASE_URL = "https://openrouter.ai/api"
     $env:ANTHROPIC_AUTH_TOKEN = "{apiKey}"
     $env:ANTHROPIC_API_KEY = ""
     ```
   - On macOS/Linux (bash/zsh syntax):
     ```
     export ANTHROPIC_BASE_URL="https://openrouter.ai/api"
     export ANTHROPIC_AUTH_TOKEN="{apiKey}"
     export ANTHROPIC_API_KEY=""
     ```
   - CRITICAL: The base URL MUST be `https://openrouter.ai/api` (no `/v1` suffix per Pitfall 1).
   - CRITICAL: `ANTHROPIC_API_KEY` MUST be set to empty string, not omitted (per Pitfall 2).
   - CRITICAL: Use `ANTHROPIC_AUTH_TOKEN` (not `ANTHROPIC_API_KEY`) for the OpenRouter key.

7. `buildAnthropicRestoreBlock(apiKey)` -- Generate a block that restores the original Anthropic key:
   - On `win32`: `$env:ANTHROPIC_API_KEY = "{apiKey}"` (only ANTHROPIC_API_KEY, no BASE_URL or AUTH_TOKEN)
   - On macOS/Linux: `export ANTHROPIC_API_KEY="{apiKey}"`
   - If `apiKey` is null/undefined (user never had one), return `null` (caller should use `removeBlock` instead).

All imports needed: `os`, `path`, `fs`, `child_process` (execSync).
